# Cloudflare staging provider capabilities

Verified against current official provider documentation on 2026-07-21. These are mutable limits and behaviours; reverify them before issues #149, #151, or #152 change production routing.

## Railway and Railpack origin contract

- Adding a Railway custom domain produces the exact CNAME and TXT ownership records that must be created; a missing TXT can leave requests returning 404, and propagation may take up to 72 hours. Railway explicitly supports Cloudflare CNAME flattening at the apex. ([Railway domains](https://docs.railway.com/networking/domains/working-with-domains))
- Current custom-domain allowances are 0 on Railway Free after trial, 1 on Trial, 2 per service on Hobby, and 20 per service on Pro. The apex and `www` therefore consume both Hobby slots on the frontend service. ([Railway pricing](https://railway.com/pricing), [Railway domain considerations](https://docs.railway.com/networking/domains/working-with-domains#important-considerations))
- Railway's current Cloudflare guidance says proxied Railway custom domains must use **Full** and warns that **Full (strict)** may fail when Cloudflare sees Railway's fallback certificate. The accepted parent contract in issue #145 nevertheless forbids weakening Full (strict), and both current Dinder custom domains report valid hostname-matching Railway certificates. Issue #148 therefore stages **Full (strict)** without proxying; issue #149 must treat the first proxied-request strict handshake as a hard gate and immediately gray-cloud on failure. ([Railway Cloudflare configuration](https://docs.railway.com/networking/domains/working-with-domains#cloudflare-configuration), [Railway SSL certificates](https://docs.railway.com/networking/domains/working-with-domains#ssl-certificates))
- Railpack detects Vite, runs the application build script, defaults static output to `dist`, and accepts `RAILPACK_SPA_OUTPUT_DIR` for a different built-static directory; setting that variable also enables SPA mode. It serves Node SPAs with Caddy and lets a root-level `Caddyfile` replace its default. ([Railpack Node static sites](https://railpack.com/languages/node/#static-sites))
- Railpack's default Caddy template supplies the static root, `/health`, SPA fallback, compression, `nosniff`, and removal of `Server`, but no cache lifetime headers. The Dinder `Caddyfile`, not a Railpack default, is therefore the origin authority for the shared HTML lifetime. ([Railpack Caddy template](https://github.com/railwayapp/railpack/blob/main/core/providers/node/Caddyfile.template), [Railway SPA configuration](https://docs.railway.com/guides/spa-routing-configuration))

## Pre-traffic Cloudflare zone

- Cloudflare Free supports only a primary/full zone. It does not become authoritative until the registrar delegates the assigned Cloudflare nameservers; a Free zone left pending is automatically deleted after 28 days. Issue #148 can stage it, but issue #149 must activate it inside that window or recreate it. ([full-zone setup](https://developers.cloudflare.com/dns/zone-setups/full-setup/), [zone status](https://developers.cloudflare.com/dns/zone-setups/reference/domain-status/))
- Cloudflare's quick scan is heuristic and can miss custom hostnames, DKIM, mail, and other records. Cloudflare says to review and manually add missing records before changing nameservers; its migration guidance says to obtain a clean, current, complete export from the existing authoritative provider and compare the import against it. ([quick-scan limitations](https://developers.cloudflare.com/dns/zone-setups/reference/dns-quick-scan/), [migration preparation](https://developers.cloudflare.com/learning-paths/dns-best-practices/concepts/phase-2/))
- BIND import/export is supported. Imports are limited to 256 KiB and three API requests per minute per user. The dashboard's **Proxy imported DNS records** option must be cleared, and per-record `cf-proxied:false` tags are stronger because `cf-proxied` takes precedence over the general import option. ([import/export](https://developers.cloudflare.com/dns/manage-dns-records/how-to/import-and-export/))
- A Free zone created after 2024-09-01 allows 200 user-created DNS records. Records created by other Cloudflare services also count. ([DNS record quota](https://developers.cloudflare.com/dns/manage-dns-records/))
- Only A, AAAA, and CNAME records are proxy-eligible; MX, TXT, and other types are always DNS-only. A verification CNAME should remain DNS-only. DNS-only TTLs on non-Enterprise plans may be 60 to 86,400 seconds, while `Auto` is 300 seconds. ([proxy status](https://developers.cloudflare.com/dns/proxy-status/), [TTL](https://developers.cloudflare.com/dns/manage-dns-records/reference/ttl/))
- DNS-only records expose the configured origin value and Cloudflare cannot optimize, cache, protect, or provide HTTP analytics for their traffic. That is the required issue #148 posture for **every** imported record, including the future frontend hosts. ([proxy status](https://developers.cloudflare.com/dns/proxy-status/), [cache start guide](https://developers.cloudflare.com/cache/get-started/))

The canonical pre-import manifest must therefore enumerate every user-managed record's owner, type, content/data, priority where applicable, intended TTL, and `proxied: false`; it must be compared with independent authoritative answers before import. No registrar nameserver or live authoritative DNS change belongs to this issue.

## Namecheap email forwarding dependency

Namecheap's free Email Forwarding service works only while the domain uses Namecheap BasicDNS, PremiumDNS, or FreeDNS; copying its MX and SPF records to another authoritative DNS provider does not preserve the forwarding control-plane service. ([Namecheap free Email Forwarding](https://www.namecheap.com/support/knowledgebase/article.aspx/308/2214/how-to-set-up-free-email-forwarding/), [Namecheap DNS dependency](https://www.namecheap.com/support/knowledgebase/article.aspx/323/46/why-cant-i-modify-email-domain-redirect-and-host-records-in-my-namecheap-account/))

The authenticated **Domain > Redirect Email** panel was inspected on 2026-07-21. It reported that no Email Redirect was defined and exposed only the actions to add a forwarder or catch-all. The redacted evidence therefore records zero active aliases and no catch-all, with no destination values present or captured. Issue #149 is not blocked by an existing forwarding dependency, but must repeat this check immediately before changing nameservers; any newly active forwarder blocks cutover until a compatible replacement and delivery test are staged.

## HTML Cache Rule on Free

Cache Rules exist on every plan, with **10 rules on Free**, but affect traffic only for proxied DNS records. The staged rule is inert while issue #148 keeps all records DNS-only. ([Cache Rules availability](https://developers.cloudflare.com/cache/how-to/cache-rules/))

Cloudflare supports hostname, request-method, and request-header fields plus `contains`/set expressions. String comparisons are case-sensitive, so normalizing `Accept` makes the intended media-type match robust. ([Cache Rule fields and operators](https://developers.cloudflare.com/cache/how-to/cache-rules/settings/), [Rules functions](https://developers.cloudflare.com/ruleset-engine/rules-language/functions/), [Rules values](https://developers.cloudflare.com/ruleset-engine/rules-language/values/))

```text
(http.host in {"dinder.it.com" "www.dinder.it.com"}
 and http.request.method in {"GET" "HEAD"}
 and any(lower(http.request.headers["accept"][*])[*] contains "text/html"))
```

The rule policy is:

| Setting                           | Required value                                                                                        | Provider basis                                                                                                                                                                                                                                                                                                                                                               |
| --------------------------------- | ----------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Cache eligibility                 | Eligible (`cache: true`)                                                                              | HTML is not cached by default; Cache Rules can make it eligible. ([default cache behaviour](https://developers.cloudflare.com/cache/concepts/default-cache-behavior/))                                                                                                                                                                                                       |
| Edge TTL                          | Use origin when present, bypass otherwise (`bypass_by_default`); no status-code or fixed-TTL override | This mode follows an origin cache header if present and does not invent a lifetime when it is absent. Free has Origin Cache Control enabled and cannot disable it. ([Cache Rule settings](https://developers.cloudflare.com/cache/how-to/cache-rules/settings/), [Origin Cache Control](https://developers.cloudflare.com/cache/concepts/cache-control/))                    |
| Query string                      | Ignore the full query string                                                                          | Free supports Ignore query string. It keeps the default-key components except the URI query, so scheme, host, and path remain. ([Cache keys](https://developers.cloudflare.com/cache/how-to/cache-keys/), [ignore-query example](https://developers.cloudflare.com/cache/how-to/cache-rules/examples/cache-everything-ignore-query-strings/))                                |
| Identity/representation variation | Add none: device type off; no custom headers, cookies, host, user, or Vary settings                   | Free cannot customize header/cookie/host/user components. The disabled checked **Include origin header** control is Cloudflare's default key, not a custom variation added by this rule. ([Cache-key availability](https://developers.cloudflare.com/cache/how-to/cache-keys/), [Cache Rule settings](https://developers.cloudflare.com/cache/how-to/cache-rules/settings/)) |
| Cache deception armor             | On                                                                                                    | Available on Free and configurable in the Cache Key setting. ([Cache-key availability](https://developers.cloudflare.com/cache/how-to/cache-keys/), [Cache Rule settings](https://developers.cloudflare.com/cache/how-to/cache-rules/settings/))                                                                                                                             |
| Serve stale while updating        | Disabled (`serve_stale.disable_stale_while_updating: true`)                                           | Cloudflare documents this exact action and boolean. ([Cache Rule settings](https://developers.cloudflare.com/cache/how-to/cache-rules/settings/), [revalidation](https://developers.cloudflare.com/cache/concepts/revalidation/))                                                                                                                                            |
| Browser TTL                       | No override                                                                                           | A Browser TTL rule would replace the downstream browser lifetime. ([CDN cache-control interactions](https://developers.cloudflare.com/cache/concepts/cdn-cache-control/))                                                                                                                                                                                                    |

`cloudflare/staging-policy.json` is a normalized record of the saved dashboard state, not a deployable Cloudflare API payload. In particular, its Free-plan Ignore Query String mode is not represented as an Enterprise `custom_key` query-string, header, cookie, host, or user component.

Cloudflare gives `Cloudflare-CDN-Cache-Control` precedence over `CDN-Cache-Control` and ordinary `Cache-Control`, and consumes the Cloudflare-specific header rather than sending it to browsers. Dinder HTML therefore emits all three layers: `Cloudflare-CDN-Cache-Control: public, max-age=60, must-revalidate` controls the intended Cloudflare lifetime; `CDN-Cache-Control: no-store` makes an accidentally missing Cloudflare-specific header fail closed to `BYPASS`; and browser-facing `Cache-Control: max-age=0, must-revalidate` remains unchanged. The Cache Rule's `bypass_by_default` mode does not invent a lifetime when these origin headers are absent. ([CDN-Cache-Control precedence](https://developers.cloudflare.com/cache/concepts/cdn-cache-control/), [Origin Cache Control](https://developers.cloudflare.com/cache/concepts/cache-control/))

Cloudflare's default cache key is not URL-only: it includes the full URL, client `Origin`, and documented method/forwarded override headers. Ignore Query String removes only the URI query. On Free, the header, cookie, host, and user custom-key sections are unavailable; the dashboard therefore shows **Include origin header** checked and disabled. The staged rule adds no custom identity or representation dimension, but later rollout evidence must not describe the resulting key as literally header-free. ([Cache keys](https://developers.cloudflare.com/cache/how-to/cache-keys/), [Cache Rule settings](https://developers.cloudflare.com/cache/how-to/cache-rules/settings/))

Cloudflare converts cacheable HEAD misses to GET at the origin and returns headers only, so including both methods is supported as long as the origin handles GET. ([HEAD cache behaviour](https://developers.cloudflare.com/cache/concepts/cache-behavior/))

## Adjacent cache and TLS settings

- **Always Online: off.** It is available on Free and can serve Cloudflare-stale or Internet Archive content when the origin is unreachable, which conflicts with the required no-archive/no-independent-stale policy. ([Always Online](https://developers.cloudflare.com/cache/how-to/always-online/))
- **Tiered Cache: off.** Tiered and Smart topology are available on Free and can be explicitly disabled; leaving them off avoids another cache layer before rollout evidence exists. ([Tiered Cache](https://developers.cloudflare.com/cache/how-to/tiered-cache/))
- **SSL/TLS mode: Full (strict).** Cloudflare recommends this mode when the origin presents a matching publicly trusted or Origin CA certificate, and the accepted parent contract requires it. Railway's current provider-specific guidance instead requires Full and warns Full (strict) may fail. Staging keeps Full (strict); no origin TLS traffic is affected while every record is DNS-only. ([Cloudflare Full strict](https://developers.cloudflare.com/ssl/origin-configuration/ssl-modes/full-strict/), [Railway Cloudflare configuration](https://docs.railway.com/networking/domains/working-with-domains#cloudflare-configuration))

The Railway guidance and issue [#145](https://github.com/Zacplischka/dinner_app/issues/145) are intentionally recorded as a live conflict, not silently reconciled. Issue #149 must require a successful Full-strict handshake on the first proxied request and gray-cloud immediately if Railway presents a certificate Cloudflare rejects.

## Production identifiers and purge credential

- Cloudflare documents zone/account IDs as reusable environment/configuration values and explicitly distinguishes non-secret IDs from the API token, which must be stored as a secret. The production zone ID may therefore be ordinary production config; the token may not. ([API environment variables](https://developers.cloudflare.com/fundamentals/api/how-to/make-api-calls/), [non-secret IDs versus token](https://developers.cloudflare.com/sandbox/api/tunnels/))
- A custom API token can be restricted to one named zone. The only permission required by the zone purge endpoint is the zone-level **Cache Purge** permission, so the production purge token needs no DNS, Cache Rules, Zone Settings, or account-wide permission. ([create token and scope resources](https://developers.cloudflare.com/fundamentals/api/get-started/create-token/), [API token permissions](https://developers.cloudflare.com/fundamentals/api/reference/permissions/), [purge endpoint](https://developers.cloudflare.com/api/resources/cache/methods/purge/))
- Free supports purge by URL, hostname, tag, prefix, and everything. The account limit for hostname/tag/prefix/everything requests is five per minute with a bucket of 25; single-file purge has separate limits. ([purge availability and limits](https://developers.cloudflare.com/cache/how-to/purge-cache/))
- The token secret is shown only once and must be treated like a password. Redacted evidence may capture its name, token ID, active status, expiry/not-before values, one-zone resource scope, and sole Cache Purge permission, but never the token value or an Authorization header. ([token creation](https://developers.cloudflare.com/fundamentals/api/get-started/create-token/), [token verification metadata](https://developers.cloudflare.com/api/resources/user/subresources/tokens/methods/verify/))

## Items not confirmable from provider documentation alone

- Provider docs cannot prove the active Railway frontend service's root directory, exact Railpack version, or live response headers; verify those from Railway state and a direct-origin response. The repository's current custom `Caddyfile` is the owned cache-header contract.
- Cloudflare does not make a blanket statement that every identifier is public. Its official configuration examples do explicitly treat account/zone IDs as non-secret variables and the API token as a secret; that is the narrow basis for the policy above.
- Staging a non-authoritative pending zone and DNS-only records cannot route current production traffic through Cloudflare by definition, but provider docs do not substitute for querying the real registrar delegation, authoritative nameservers, record set, and Cloudflare zone state immediately before and after staging.
